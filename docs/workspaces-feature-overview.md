# CorpMeet — описание функционала после внедрения пространств

## Краткая суть

Сейчас CorpMeet — это бот и веб-приложение для бронирования **одной общей переговорной** одной команды, привязанной к одному Telegram-чату. Все, кто зарегистрировался, попадают в общий пул и видят расписание этой одной комнаты.

После доработки система превращается в **многотенантную платформу**: появляются «Пространства» (компании/команды) и «Комнаты» (физические переговорные). Одна и та же платформа теперь обслуживает много разных компаний, а одна и та же физическая комната может бронироваться сразу несколькими компаниями (например, разные команды в одном бизнес-центре делят переговорки).

## Кто такие пользователи

Каждый пользователь — это аккаунт CorpMeet, привязанный к Telegram. Пользователь может состоять в нескольких пространствах одновременно и переключаться между ними (как воркспейсы в Slack или Notion).

Внутри каждого пространства у пользователя одна из трёх ролей:

- **Owner** — создатель пространства. Один на пространство. Может всё, включая удаление пространства и передачу владения.
- **Admin** — назначается owner-ом. Управляет участниками, комнатами, бронированиями всех участников.
- **Member** — обычный участник. Бронирует, видит общий календарь, никем не управляет.

Над пространствами есть глобальная роль **superadmin платформы** — для операторов сервиса, доступ ко всему.

## Регистрация и онбординг

Сейчас при первом входе пользователь либо проходит в систему (если он член «той самой» Telegram-группы), либо получает отказ. После доработки отказа быть не должно — вместо него появляется экран развилки с тремя путями:

1. **«Создать своё пространство»** — пользователь вводит название, становится owner-ом, получает уникальный invite-код для приглашения других. Опционально привязывает к пространству свою Telegram-группу — туда будут уходить уведомления.
2. **«Войти по коду»** — пользователь вводит invite-код пространства, отправляется заявка админам этого пространства. Они видят запрос в личных уведомлениях и в TG-группе пространства, аппрувят или отклоняют.
3. **«Найти по названию»** — пользователь начинает набирать название пространства, видит автодополнение, выбирает нужное, отправляется заявка.

Этот же экран показывает бот в чате — три inline-кнопки после `/start`.

## Пространство — что это

Пространство (workspace) — это контейнер для команды/компании со своими настройками:

- название и slug (для URL и поиска),
- уникальный invite-код для приглашений,
- привязанная Telegram-группа (опционально),
- часовой пояс,
- список участников с ролями,
- список комнат, к которым у пространства есть доступ.

В шапке приложения появляется селектор пространств: пользователь видит список своих пространств и переключается между ними. Все данные (календарь, комнаты, участники) фильтруются по выбранному пространству.

### Управление пространством

Owner и admin видят страницу настроек пространства с разделами:

- **Участники** — список активных, отдельная вкладка с pending-заявками. Действия: пригласить по `@username`, сделать админом, отозвать админку, удалить из пространства, передать владение.
- **Комнаты** — список «Мои комнаты» (где пространство — владелец) и «Подключённые» (расшарены извне). Действия зависят от роли в комнате.
- **Общее** — название, invite-код с кнопкой regenerate, инструкция по привязке Telegram-группы.

### Приглашение по @username

Owner или admin вписывает `@username`. Система находит пользователя в БД, создаёт `pending`-членство и через бота отправляет в Telegram сообщение «Вас пригласили в пространство X. Принять / отклонить». Если такого пользователя в БД ещё нет, приглашение «висит» по username и автоматически активируется при первом `/start` бота.

### Заявка на вступление

Когда новый пользователь подаёт заявку (по коду или по названию), её видят owner и admin в pending-вкладке. Решение — «принять» или «отклонить». Бот может дублировать заявки сообщением в TG-группу пространства, если она привязана.

## Комнаты — общий физический ресурс

Это самое важное отличие от наивной модели «комната принадлежит компании».

**Комната — это физический ресурс**, не привязанный к конкретному пространству. Одна и та же комната может быть подключена к нескольким пространствам — типичная ситуация, когда несколько компаний в одном здании пользуются одними и теми же переговорками.

### Кто создаёт

Создавать новые комнаты могут **только admin и owner** какого-либо пространства. Member кнопки «+ Новая комната» вообще не видит, и попытка POST на API возвращает 403.

При создании комнаты автор автоматически получает к ней доступ от имени своего пространства с ролью **owner-комнаты**. Это значит, что его пространство может её редактировать, удалять, расшаривать с другими, отзывать доступ.

### Кто бронирует

**Любой участник** пространства (включая member) выбирает комнату из списка тех, к которым у его текущего пространства есть доступ. Никаких прав на создание комнат для этого не нужно — просто выбирай и бронируй.

### Шеринг

Admin/owner пространства, владеющего комнатой, может расшарить её с другим пространством, указав его slug или invite-код. Принимающее пространство получает роль **shared** на эту комнату — может видеть её и бронировать, но не может редактировать, удалять или расшаривать дальше.

В любой момент владелец комнаты может отозвать доступ. Принимающая сторона тоже может «отключить» комнату у себя.

### Пересечения по времени — глобальные

Это ключевой пункт: проверка пересечений идёт **по физической комнате независимо от пространства**. Если пространство А забронировало комнату на 14:00–15:00, пространство Б на это же время в эту же комнату забронировать не сможет — получит ошибку «Время пересекается».

### Режим видимости (visibility)

У каждой комнаты настраиваемый режим показа чужих бронирований:

- **full** (по умолчанию) — все пространства, подключённые к комнате, видят все детали: название встречи, организатора, описание, гостей. Подходит для случая «офис, где все друг друга знают».
- **busy_only** — пространство видит только свои бронирования полностью; чужие отображаются как серая плашка «Занято» с временем, без названия, организатора и описания. Подходит для случая «несколько независимых компаний в общем коворкинге».

В календаре чужие бронирования с `busy_only` показываются заглушками — чтобы пользователь видел, что слот занят, но не знал чем.

### Удаление и архивация

Если admin удаляет комнату (soft-delete), это запрещено пока есть будущие бронирования из любого пространства. Если пространство-владелец комнаты удаляется — права на эту комнату наследуются первому пространству с shared-доступом. Если других нет — комната тоже архивируется.

## Бронирования

С точки зрения пользователя процесс почти не меняется: открыл календарь, кликнул в свободный слот, ввёл название, выбрал комнату из списка, добавил гостей по `@username`, сохранил.

Что внутри:

- Каждое бронирование привязано к **двум сущностям**: пространство (от чьего имени создано) и комната (физический ресурс).
- Пространство нужно для определения, куда слать уведомления (в TG-группу того пространства, не владельца комнаты).
- Комната нужна для проверки пересечений и для группировки в календаре.
- Гости из поля `guests` (список `@username`) теперь резолвятся только среди членов того же пространства — нельзя добавить гостем участника чужой компании.
- Если пользователь удалён из пространства, его будущие бронирования остаются в силе, но новых создать нельзя.

## Telegram-уведомления

Каждое пространство опционально привязывает свой Telegram-чат. Привязка делается командой `/bind <invite_code>` в нужной группе (вызвавший должен быть owner или admin пространства).

Бот шлёт уведомления о бронированиях, изменениях и напоминаниях в TG-группу того пространства, **от которого создано бронирование**, а не в группу владельца комнаты. Это значит: если две компании делят одну комнату, у каждой свой канал уведомлений, и они друг друга не спамят. Если у пространства группа не привязана — групповые уведомления пропускаются, остаются только личные сообщения организатору и гостям.

## Что увидит существующий пользователь после миграции

Все текущие пользователи автоматически попадают в дефолтное пространство «CorpMeet», единственная сейчас переговорная становится первой глобальной комнатой, доступной этому пространству, привязка к старой Telegram-группе сохраняется. Никто ничего не теряет, всё продолжает работать. После апдейта существующие админы получают возможность создавать новые комнаты, приглашать новых пользователей по коду, при желании — расшаривать комнаты с другими пространствами.

## Что увидит новый пользователь после миграции

После регистрации — экран развилки. Дальше один из трёх путей:

- создал своё пространство → попал в пустой календарь → создал свою первую комнату → бронирует;
- ввёл код чужого пространства → ждёт аппрува → попал внутрь → видит общий календарь и список доступных комнат → бронирует;
- нашёл пространство по названию → отправил заявку → дальше так же.

## Сценарий «общий бизнес-центр»

Иллюстрация того, ради чего вся доработка:

В одном здании работают три компании: «Альфа», «Бета» и «Гамма». На этаже три переговорки: «Москва», «Лондон», «Токио». Сценарий:

- Сотрудник «Альфы» регистрируется первым, создаёт пространство «Альфа» и три комнаты «Москва», «Лондон», «Токио». Его пространство — owner всех трёх комнат.
- Приходит сотрудник «Беты», создаёт пространство «Бета». Договариваются с «Альфой» — admin «Альфы» расшаривает все три комнаты пространству «Бета» (по invite-коду «Беты»). Теперь у «Беты» доступ ко всем трём комнатам с ролью shared.
- «Бета» бронирует «Москву» на 14:00–15:00. «Альфа» в это же время бронирует «Москву» — получает 409, время занято.
- В режиме `busy_only` сотрудник «Беты» видит в календаре, что 14:00–15:00 «Москва» занята, но не видит ни кто её занял, ни зачем.
- Уведомление о бронировании уходит в Telegram-группу «Беты», в группу «Альфы» — нет.
- Через месяц приходит «Гамма», работает в своих кабинетах, ничего не расшаривают — у них своё пространство, своя приватная комната, никого не видят и сами невидимы.

Это и есть базовый паттерн использования.
